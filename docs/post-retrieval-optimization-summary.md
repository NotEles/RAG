# Post-Retrieval Optimization Overview

本文档总结当前分支对 RAG 检索后优化能力的整体建设。这里的“检索后优化”指向量检索完成之后、生成模型读取上下文之前，对候选片段进行去重、重排、压缩、多样性筛选和上下文打包的一组处理流程。

本次改造的核心目标不是单独增加某一个算法，而是把原本松散的检索结果处理整理成一条可配置、可观察、可回退的后处理链路，使 Search、QA 和 Generation 三个主要使用场景都能共享同一套优化能力。

## Goals

本次改造主要解决以下问题：

- 检索结果直接进入生成阶段时，容易包含重复片段、弱相关片段或超出上下文预算的长文本。
- 原始向量相似度不足以表达最终可回答性，尤其在多查询、父子块、融合召回和长文档场景下，需要二次排序。
- 压缩上下文如果完全依赖生成模型自由发挥，容易出现信息丢失或不可靠改写，需要保守校验和失败回退。
- 用户需要知道最终上下文经历了哪些优化步骤，否则很难判断回答质量问题来自召回、重排、压缩还是生成。
- Search、QA 和 Generation 页面过去对后处理参数和状态的支持不一致，导致同一能力在不同入口下表现不同。

## Overall Design

当前实现将检索后优化抽象为一条可组合 pipeline。一次请求可以按需启用以下策略：

- `deduplicate`：去除重复或高度重合的候选片段。
- `rerank`：对候选片段重新排序，优先保留更能回答问题的内容。
- `compress`：压缩长片段，减少无关文本占用上下文窗口。
- `diversify`：在相关性和来源多样性之间做折中。
- `context_pack`：按照字符和 token 预算打包最终上下文。

这条 pipeline 的输入是检索服务返回的候选结果，输出仍保持为 RAG 生成阶段可以直接消费的结果列表。这样做的好处是：优化能力不会侵入底层向量数据库，也不会绑定某一个生成模型，后续可以继续替换或扩展具体策略。

## Retrieval Candidate Handling

为了让重排和多样性筛选有足够候选，本次改造增加了候选召回控制：

- 当启用重排或多样性筛选时，系统会先召回比最终返回数更多的候选片段。
- 候选召回阈值可以比最终阈值更宽松，避免高质量片段在重排前就被过滤掉。
- 后处理完成后再应用最终阈值和最终数量限制，保证返回结果仍符合用户预期。

这种方式让“召回阶段”和“精选阶段”各司其职：向量检索负责尽量找齐可能相关的内容，检索后优化负责排序、裁剪和打包。

## Reranking

重排支持三类路径：

- 本地 CrossEncoder 重排：适合速度优先和离线场景。
- 本地 Ollama LLM 重排：默认可使用 `qwen2.5:3b`，适合更强调语义判断的场景。
- 词面相关性降级：当模型不可用、加载失败或 LLM 返回不可解析时，系统不会中断请求，而是降级到轻量相关性评分。

重排后的结果会保留原始召回分数、重排分数和最终分数，便于前端展示和后续分析。对于融合召回结果，最终分数会综合融合分数和重排分数，避免完全丢弃原始检索信号。

## Compression

压缩分为抽取式压缩和 LLM 压缩。

抽取式压缩会从原始片段中选择与问题更相关的句子，优点是速度快、可控性强、不会引入原文之外的新信息。

LLM 压缩更适合处理长片段和语义密度较高的文本。为控制成本和延迟，前端可以配置只对前 N 条结果执行 LLM 压缩。压缩结果会经过多层校验：

- 输出为空或长度不合理时拒绝使用。
- 压缩结果过短时拒绝使用。
- 如果原文包含查询关键词，而压缩结果完全丢失这些关键词，会拒绝使用。
- 如果压缩结果与原文词面重合过低，会拒绝使用。
- 对跨语言或低词面重合场景，会额外让 LLM 判断压缩片段是否保留核心信息。

如果压缩失败或被拒绝，系统默认保留原始片段，而不是直接删除上下文。只有在显式允许删除无关片段、且相关性足够低时，才会丢弃片段。这是为了让检索后优化保持保守可靠，不因为一次不稳定的 LLM 输出损坏最终答案。

## Context Packing

上下文打包负责将优化后的片段放入最终上下文预算中。它同时考虑字符数和 token 数：

- 未超出预算的片段会按当前顺序加入上下文。
- 超出预算的片段会尽量按句子边界截断。
- 如果句子级截断仍不能满足预算，会退回到字符级截断。
- 每个被打包的片段会记录顺序和估算 token 数。

这使最终进入生成模型的上下文更可控，尤其适合本地小模型或上下文窗口有限的模型。

## Frontend Experience

Search、QA 和 Generation 页面都增加了检索后优化配置。

用户可以选择预设：

- 质量优先：启用去重、LLM 重排、LLM 压缩和上下文打包。
- 速度优先：启用去重、本地重排和上下文打包。
- 本地离线：尽量使用本地能力完成重排和压缩。

用户也可以手动调整：

- 候选召回数量。
- 重排后保留数量。
- 重排方式和重排模型。
- 压缩方式。
- LLM provider、模型和 API Key。
- LLM 压缩前 N 条。
- MMR 相关性权重。
- 字符预算和 token 预算。

Generation 页面支持对已保存或已传入的搜索结果再次执行后处理。为了避免重复处理已优化结果，页面会识别已有后处理痕迹，并提供“重新优化已有后处理结果”的开关。

## Trace and Observability

本次改造新增了 `postprocess_trace`，用于记录每个后处理步骤的输入数量、输出数量、耗时和关键配置。

Trace 会在以下场景中返回或保存：

- `/search` 响应。
- `/qa` 响应。
- `/generate` 响应。
- 保存搜索结果文件。
- Generation 页面加载保存结果。

前端新增统一的 Trace 组件，用来展示：

- 标准化。
- 去重。
- 重排。
- 阈值过滤。
- 压缩。
- 多样性筛选。
- 上下文打包。
- 最终数量截断。
- LLM 压缩的成功、跳过、拒绝、失败和删除统计。

这让用户可以直接看到某次请求的上下文是如何从原始候选一步步变成最终输入的，也方便后续定位质量问题。

## Persistence and Evaluation

保存搜索结果时会一并保存后处理 Trace。这样从保存文件进入 Generation 页面时，用户不仅能看到结果本身，也能看到这些结果曾经经历过的优化链路。

此外新增了评估脚本，用于对比不同后处理配置的效果，例如：

- baseline 检索结果。
- 抽取式后处理。
- 本地 Ollama 重排和压缩。

评估脚本的目的不是替代人工质量判断，而是提供一个可重复的实验入口，方便后续比较不同参数、模型和策略组合。

## Reliability Choices

本次实现采取了偏保守的可靠性策略：

- 后处理默认不改变未启用策略的行为。
- 模型调用失败时不让请求整体失败，而是降级或保留原结果。
- LLM 压缩默认不删除无关片段，除非显式允许。
- `/generate` 不再默认按重排数量截断上下文，避免用户在生成页无意丢失已加载的证据。
- Search、QA 和 Generation 使用同一套后处理配置结构，降低入口之间的行为差异。
- Trace 不依赖服务实例上的共享状态，避免并发请求之间串扰。

这些选择的目标是让检索后优化成为“质量增强层”，而不是新的不稳定来源。

## Current Limitations

当前实现仍有一些值得后续继续优化的地方：

- Generation 页面判断“是否已经优化过”的逻辑仍偏粗，目前主要依据结果中是否存在后处理痕迹。更理想的方式是按策略逐项判断，只有当前选择的策略都已执行过时才跳过。
- CrossEncoder 模型首次加载会带来延迟，可以考虑增加预热或模型状态提示。
- LLM 压缩质量仍依赖模型输出稳定性，后续可以增加更系统的压缩质量评估指标。
- 当前 Trace 主要展示步骤级统计，未来可以增加更细的单片段级解释视图。
- 多样性筛选目前是轻量实现，后续可以接入 embedding 相似度以提升语义层面的多样性判断。
