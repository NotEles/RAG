# 文件解析优化 —— 修改总结

在原有 RAG 系统 PDF 智能解析能力的基础上，新增通用文件解析入口，使系统能够支持 JSON、DOCX、Markdown、TXT、CSV、PNG/JPG 图片等多种格式输入。所有新增格式都会被统一转换为后续 RAG 流程可使用的文本块结构，兼容原有加载、分块、向量化、入库和问答流程。

### 向后兼容设计

- PDF 原有解析方式全部保留：PyMuPDF、PyPDF、PDFPlumber、Unstructured
- 非 PDF 文件无需用户关心加载工具，后端会根据扩展名自动选择解析器
- 原有 `/load`、`/parse`、`/process`、`/import-to-course` 流程继续可用
- `parse_pdf()` 保留兼容包装，新逻辑统一走 `parse_document()`

## 二、修改 `backend/services/loading_service.py`

新增通用文档加载方法 `load_document()`，作为多格式解析的统一入口。

### 支持格式

| 文件格式 | 处理方式 | 依赖 |
|------|------|------|
| PDF | 复用 PyMuPDF、PyPDF、PDFPlumber、Unstructured | PyMuPDF / pypdf / pdfplumber / unstructured |
| JSON | 自动提取结构化文本，兼容 chunks 和问答格式 | Python 标准库 json |
| JSONL | 按行读取 JSON 对象并转换为文本 | Python 标准库 json |
| Markdown | 读取 Markdown 文本，并按标题分段 | 无额外依赖 |
| TXT | 直接读取纯文本 | 无额外依赖 |
| CSV | 按行提取字段内容 | pandas |
| DOCX | 提取段落、标题和表格文本 | python-docx |
| PNG/JPG 等图片 | OCR 识别图片文字 | pillow + pytesseract + 系统 Tesseract |

### 核心逻辑

```python
def load_document(self, file_path, method="auto", ...):
    suffix = Path(file_path).suffix.lower()

    if suffix == ".pdf":
        return self.load_pdf(...)
    if suffix in {".json", ".jsonl"}:
        return self._load_json(file_path)
    if suffix in {".md", ".markdown"}:
        return self._load_markdown(file_path)
    if suffix == ".docx":
        return self._load_docx(file_path)
    if suffix in image_suffixes:
        return self._load_image(file_path)
```

所有解析器最终都会生成统一的 `current_page_map`：

```python
[
  {
    "page": 1,
    "text": "...",
    "metadata": {...}
  }
]
```

这样后续分块、embedding 和入库逻辑无需针对不同文件类型重复开发。

## 三、修改 `backend/services/parsing_service.py`

将原先面向 PDF 的解析入口抽象为通用文档解析入口。

### 新增方法

```python
def parse_document(self, text, method, metadata, page_map=None):
    ...
```

### 保留兼容方法

```python
def parse_pdf(self, text, method, metadata, page_map=None):
    return self.parse_document(text, method, metadata, page_map=page_map)
```

这样既能支持新格式，又不会破坏旧的 PDF 调用路径。

## 四、修改 `backend/main.py`

将多个接口中的 `load_pdf()` 调用替换为 `load_document()`。

### 受影响接口

| 接口 | 修改内容 |
|------|------|
| `/process` | 上传文件后使用通用加载入口 |
| `/parse` | 文件解析流程支持非 PDF |
| `/load` | 文档读入页面支持多格式保存 |
| `/import-to-course` | 一键导入 course 支持多格式文件 |
| `/chunk` | 输出文件名去除 PDF 固定假设 |

### `/import-to-course` 优化

原流程只支持 PDF / JSON，现在扩展为：

```text
上传文件 -> load_document() -> chunk_text() -> create_embeddings() -> add_to_named_collection("course")
```

并保留 JSON 问答格式的特殊处理逻辑：

- 已有 `chunks` 字段：直接使用
- 含 `question` + `answers`：按问答对生成 chunk
- 普通 JSON：自动抽取文本后进入统一分块流程

## 六、修改 `backend/services/vector_store_service.py`

优化 `course` 知识库状态展示。

### 问题

前端“最近文件信息”原来通过：

```python
collection.get(limit=1)
```

获取第一条 metadata，因此多次导入后仍可能显示最早导入的文件。

### 修复

改为读取一批 metadata，并按 `embedding_timestamp` 选择最新记录：

```python
raw = max(
    metadatas,
    key=lambda item: str(item.get("embedding_timestamp", ""))
)
```

这样前端刷新或导入完成后可以展示最新导入文件。

## 十、总结

本次文件解析优化将原本以 PDF 为中心的 RAG 输入流程扩展为通用文档输入流程。新增的 `load_document()` 统一屏蔽不同文件格式的解析差异，使 JSON、DOCX、Markdown、图片等内容都能复用原有的分块、embedding、向量库和问答能力。整体改动保持向后兼容，原 PDF 解析策略不受影响，同时显著提升了系统对真实业务文档的适配能力。
